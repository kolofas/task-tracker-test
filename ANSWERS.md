### Конкурентность

Два пользователя одновременно пытаются сменить статус одной задачи:

— Пользователь A: `in_progress` -> `review`

— Пользователь B: `in_progress` -> `created`

Оба читают текущий статус `in_progress`, оба прошли валидацию. Что произойдёт? Как бы ты это решил?
#### Ответ
```
Если два пользователя одновременно меняют статус, то без защиты будет "гонка данных": оба провалидируют in_progress, и победит последний update, в итоге, данные могут расходится с действительностью. Я бы решил это блокировкой строки в транзакции.


начали транзакцию
SELECT task FOR UPDATE (в этот момент база блокирует строку и другая транзакция будет ждать)
проверили статус
изменили статус
commit()
```

### Дебаг по логам

Приходит баг-репорт: "Задача #4521 застряла в статусе `review`, хотя я точно нажимал кнопку 'Готово'".

Какие шаги ты предпримешь для расследования? Какие логи/данные посмотришь? Какие гипотезы проверишь первыми?

#### Ответ

```
Сначала я проверю текущее состояние задачи в базе и ее историю статусов. Затем буду смотреть логи приложения, чтобы убедиться, приходил ли запрос на изменение статуса и не было ли ошибок. Если запрос был, то проверю валидацию и возможные гонки данных. Если запрос нет, то смотрю инфраструктурные логи (например, nginx), чтобы понять, дошел ли запрос до сервера
```

### Проектирование

Завтра приходит продукт и говорит: "Нам нужно добавить комментарии к задачам, и чтобы при смене статуса можно было оставить комментарий-причину".

Как бы ты это реализовал? Что поменяешь в текущей схеме? Нужна ли отдельная таблица?

#### Ответ
```
Я бы добавил поле comment task_status_history, чтобы сохранять причину смены статуса вместе с событием перехода. Для обычных комментариев сделал бы отдельную таблицу task_comments, поскольку это отдельная сущность: комментарии могут появиться в любое время, их может быть много, и они не всегда связаны со сменой статуса. Далее, обновление моделей, миграции и проверка эндпоинтов
```

Кэшировал бы не сами задачи вообще, а результаты популярных запросов списка. То есть я бы шел от замеров и конкретного "узкого места"
